# Product Requirement Documents

A PRD defines *what* we're building and *how we'll know it works* — the problem, the user, the scope, the acceptance criteria, and the kill-criterion. It does **not** prescribe implementation (that's code + ADRs).

Every PRD must answer: **what is the painkiller, who feels the pain, what's the aha demo, what metric decides success, and what kills it?** (The five questions the strategy paper asks of every product.)
